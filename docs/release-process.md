# Версии и публикация

Документ соответствует версии `1.0.4`.

## Репозиторий и сайт

- репозиторий: `48rn2krftk-dev/fuel-rate-calc`;
- основная ветка: `main`;
- сайт: `https://48rn2krftk-dev.github.io/fuel-rate-calc/`;
- workflow: `.github/workflows/deploy-pages.yml`.

Vite собирает приложение с базовым путём `/fuel-rate-calc/`.

## Разделение разработки и публикации

Обычный push в `main` не публикует приложение, если не изменён
`.github/release-version`.

Deployment запускается:

1. автоматически при изменении `.github/release-version` в `main`;
2. вручную через `workflow_dispatch`.

Это позволяет хранить готовые промежуточные изменения в репозитории без
немедленного обновления приложения у пользователей.

## Где хранится номер версии

Перед релизом номер должен совпадать в четырёх местах:

| Файл | Значение для версии 1.0.4 |
| --- | --- |
| `package.json` | `"version": "1.0.4"` |
| `package-lock.json` | корневые версии `1.0.4` |
| `src/content.ts` | `APP_VERSION = "1.0.4"` |
| `.github/release-version` | `v1.0.4` |

При изменении PWA-ресурсов или поведения приложения также нужно увеличить имя
кэша в `public/sw.js`, например `hot-idle-v5`. Иначе установленное приложение
может продолжить использовать предыдущий комплект ресурсов.

## Подготовка релиза

1. Убедиться, что рабочая папка содержит только изменения релиза.
2. Обновить номера версии.
3. Обновить имя PWA-кэша при изменении пользовательской сборки.
4. Запустить:

```powershell
npm test
npm run lint
npm run build
```

5. Проверить `git diff --check`.
6. Просмотреть итоговый diff и убедиться, что случайные файлы не попали в
   релиз.

## Коммит и тег

Релизный коммит создаётся в `main`. После отправки создаётся аннотированный тег:

```powershell
git tag -a v1.0.4 -m "Release v1.0.4"
git push origin v1.0.4
```

Тег нужен для истории версий, но сам по себе не запускает текущий workflow.
Deployment запускает изменение `.github/release-version` или ручная команда
GitHub Actions.

Нельзя перемещать уже опубликованный тег на другой коммит. Для исправления
следует выпустить новую patch-версию.

## Автоматический deployment

Workflow реагирует только на:

```yaml
push:
  branches: ["main"]
  paths: [".github/release-version"]
```

Этапы:

1. checkout `main`;
2. установка LTS-версии Node.js;
3. `npm ci`;
4. `npm run build`;
5. настройка GitHub Pages;
6. загрузка каталога `dist`;
7. deployment Pages.

Workflow использует минимальные разрешения:

- чтение содержимого репозитория;
- запись GitHub Pages;
- выдача OIDC-токена для deployment.

Одновременно выполняется только один deployment. Новый запуск отменяет
незавершённый предыдущий.

## Ручной deployment

`workflow_dispatch` принимает строковое поле `version`. Сейчас оно используется
как пояснение запуска, но сборка всегда выполняется из выбранной ветки, обычно
`main`.

Ручной запуск полезен, если:

- маркер версии уже был изменён более ранним коммитом;
- автоматический запуск не произошёл;
- нужно повторить публикацию того же коммита после временной ошибки GitHub.

Ручной deployment не заменяет создание Git-тега.

## Проверка после публикации

Зелёного статуса workflow недостаточно. Нужно проверить:

1. сайт открывается по опубликованному адресу;
2. HTML ссылается на новый хешированный JavaScript и CSS;
3. экран «Ещё» показывает ожидаемую версию;
4. `sw.js` содержит ожидаемое имя кэша;
5. после одного запуска с интернетом приложение открывается без сети;
6. индикатор меняется между «Онлайн» и «Офлайн».

На iPhone обновление установленной PWA может занять время:

1. открыть сайт или приложение с интернетом;
2. дождаться установки нового service worker;
3. полностью закрыть приложение;
4. открыть его снова.

Если версия не обновилась, сначала следует проверить фактические файлы GitHub
Pages и состояние workflow, а не удалять данные пользователя.

## Роли команды

### Ведущий разработчик

- анализирует задачу;
- меняет прикладной код;
- запускает локальные проверки;
- передаёт готовый состав релиза.

### Тестирование и качество

- проверяет сценарии и регрессии;
- не меняет код без отдельного поручения;
- сообщает шаги воспроизведения, фактический и ожидаемый результат.

### Релизы и GitHub

- проверяет состав релиза;
- создаёт коммит и тег;
- отправляет изменения;
- контролирует GitHub Actions;
- проверяет опубликованные файлы.

### Техническая документация

- сверяет документы с актуальным `main`;
- описывает фактическое поведение;
- не меняет прикладной код и релизные настройки;
- обновляет документы вместе с изменением соответствующего поведения.
