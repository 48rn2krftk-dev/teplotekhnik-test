# Архитектура приложения

Документ соответствует версии `1.0.4`.

## Назначение

«Уголок теплотехника» — мобильное PWA-приложение для восстановления,
проверки и корректировки топливных цепочек, включая расчёт удельного расхода
топлива при горячем простое и помощи в устранении разрывов топливных цепочек.

Приложение работает без собственного сервера и базы данных. Расчёты выполняются
в браузере, а пользовательские данные хранятся локально на устройстве.

## Технологии

- React и TypeScript;
- Vite для локальной разработки и production-сборки;
- CSS без отдельного UI-фреймворка;
- `lucide-react` для интерфейсных иконок;
- Local Storage для настроек, слотов и истории;
- Service Worker и Web App Manifest для PWA и автономной работы;
- GitHub Actions и GitHub Pages для публикации.

## Точка входа и навигация

`src/main.tsx` создаёт React-приложение и в production регистрирует
`public/sw.js`.

`src/App.tsx`:

- хранит выбранный экран;
- переключает четыре раздела нижней навигации;
- передаёт выбранную запись истории на нужный расчётный экран;
- показывает состояние доступа к опубликованному сайту.

Доступные экраны:

| Экран | Компонент | Назначение |
| --- | --- | --- |
| ТХУ-3 | `ByTimeScreen` | Расчёт по времени начала/окончания и остаткам топлива из ведомости ТХУ-3 |
| Быстрый расчёт | `QuickScreen` | Расчёт по готовой длительности и расходу топлива |
| Сумма | `SummaryScreen` | Сложение нескольких прогревов |
| Ещё | `SettingsScreen` | Норматив, тема, слоты, история, обратная связь, поддержка и версия |

Навигация не использует URL-маршруты. Текущий экран хранится в состоянии
`App`, поэтому перезагрузка всегда открывает экран «ТХУ-3».

## Расчётные экраны

### Расчёт по ведомости ТХУ-3

`src/screens/ByTimeScreen.tsx` принимает:

- начало прогрева;
- окончание прогрева;
- топливо при приёмке;
- топливо при сдаче.

Экран управляет полями формы и подтверждением перехода на следующие сутки.
Разбор времени и даты, форматирование введённых значений и определение
длительности вынесены в `src/utils/dateTime.ts`. Полученный результат экран
передаёт в общие компоненты сравнения с нормативом и сохранения.

Одинаковое время начала и окончания без подтверждения не даёт результата. После
подтверждения следующих суток период считается равным 24 часам.

### Быстрый расчёт

`src/screens/QuickScreen.tsx` принимает готовую длительность и общий расход
топлива. Разбор длительности вынесен в `src/utils/duration.ts`.

### Сложение прогревов

`src/screens/SummaryScreen.tsx` создаёт цепочку из двух и более строк. Каждая
строка может быть:

- введена вручную;
- загружена из временного слота.

Итог появляется только после заполнения всех строк. Если задано начальное
топливо и общий расход превышает его, итоговые значения скрываются, а
сохранение блокируется.

При выборе в первой строке слота, созданного на экране «ТХУ-3», начальное
топливо цепочки заполняется из исходных данных этого слота.

## Общие компоненты

### Карточка результата

`src/components/CalculationResultCard.tsx` отображает длительность, общий и
удельный расход. Экраны передают в неё готовый результат, подписи и
дополнительные блоки сравнения или сохранения.

### Сравнение с нормативом

`src/components/NormComparison.tsx` получает готовый результат и норматив.
Компонент отвечает за:

- процент фактического расхода от нормы;
- резерв или перерасход в килограммах;
- нормативный расход за период;
- расчётный остаток топлива при сдаче.

Компонент не отображается, если норматив отсутствует или не положителен.

### Сохранение результата

`src/components/SaveResultPanel.tsx` открывает диалог сохранения. Одна операция
одновременно:

1. записывает результат в выбранный временный слот;
2. добавляет отдельную запись в постоянную историю.

Название можно изменить перед сохранением. При пустом названии используется
название текущего типа расчёта.

## Типы данных

Основные типы определены в `src/types.ts`.

`CalculationResult` содержит:

- длительность в минутах;
- общий расход топлива;
- средний расход в час.

`CalculationSource` хранит исходные данные одного из трёх типов расчёта. Эти
данные позволяют восстановить форму из истории.

`SlotData` содержит результат, название, время сохранения и, для новых записей,
исходные данные.

`HistoryEntry` дополнительно хранит:

- уникальный идентификатор;
- дату создания;
- норматив, действовавший в момент сохранения;
- обязательные исходные данные.

`AppSettings` содержит норматив и выбранную тему:

- `system` — оформление следует настройкам устройства;
- `light` — принудительно используется светлая тема;
- `dark` — принудительно используется тёмная тема.

Для новых и ранее сохранённых настроек без корректного значения темы
используется `system`.

## Локальное хранение

Логика находится в `src/utils/storage.ts`.

| Ключ Local Storage | Содержимое |
| --- | --- |
| `hotIdle.settings` | Норматив и выбранная тема |
| `hotIdle.slots` | Три временных слота |
| `hotIdle.history` | До 50 записей истории |

Слоты всегда нормализуются до массива из трёх элементов. После удаления
оставшиеся записи сдвигаются вперёд, а пустое место добавляется в конец.

История хранится от новых записей к старым. После достижения лимита старые
записи отбрасываются.

Изменения Local Storage рассылаются внутри текущей вкладки через пользовательские
события:

- `hotIdle.settingsChanged`;
- `hotIdle.slotsChanged`;
- `hotIdle.historyChanged`.

Синхронизации между устройствами, аккаунтами или браузерами нет. Очистка данных
Safari или удаление данных сайта удалит настройки, слоты и историю.

## Оформление

Логика темы находится в `src/utils/theme.ts`.

`src/main.tsx` применяет сохранённую тему до создания React-приложения.
`App.tsx` повторно применяет её при изменении настроек. Для системной темы
приложение следит за `prefers-color-scheme` и переключается между светлым и
тёмным оформлением вместе с устройством.

Фактически выбранная тема записывается в атрибут `data-theme` корневого элемента
документа. Одновременно задаётся CSS-свойство `color-scheme`.

## Обратная связь и поддержка

На экране «Ещё» доступны пользовательские каналы связи:

- e-mail;
- Telegram;
- eXpress.

Для них не требуется аккаунт GitHub. Отдельная ссылка «Яндекс Чаевые»
предназначена для добровольной поддержки разработчика.

GitHub остаётся технической ссылкой на профиль и репозиторий разработчика, а не
обязательным каналом обратной связи.

## Восстановление из истории

Нажатие на запись истории передаёт её в `App`, который открывает экран,
соответствующий `entry.source.type`.

- расчёт по ведомости ТХУ-3 восстанавливает время и оба остатка топлива;
- быстрый расчёт восстанавливает длительность и расход;
- сложение прогревов восстанавливает начальное топливо и строки как ручной ввод.

Слотовая привязка при восстановлении сложения прогревов не сохраняется:
восстанавливаются
сами числовые данные и названия элементов истории.

## PWA и автономная работа

`public/manifest.webmanifest` задаёт standalone-режим, портретную ориентацию,
цвета и иконки.

Тестовая PWA использует собственный префикс кэша `teplotekhnik-test-` и при
обновлении удаляет только свои устаревшие кэши.

При установке service worker:

1. загружает HTML приложения без использования HTTP-кэша;
2. извлекает из HTML ссылки на текущие JavaScript и CSS;
3. сохраняет HTML, сборочные ресурсы, manifest и иконки;
4. после активации удаляет предыдущие кэши приложения.

Стратегия загрузки:

- переходы между страницами сначала получают сохранённый HTML;
- статические ресурсы сначала ищутся в кэше;
- отсутствующий ресурс загружается из сети и сохраняется.

При первом посещении интернет необходим для установки service worker и
заполнения кэша. После этого приложение может запускаться без сети.

## Индикатор подключения

Индикатор в `App.tsx` не полагается только на `navigator.onLine`. Он выполняет
`HEAD`-запрос к опубликованному `manifest.webmanifest`:

- при запуске;
- каждые 15 секунд;
- после события `online`;
- при возврате фокуса или видимости приложения.

Запрос получает уникальный параметр времени, использует `cache: "no-store"` и
прерывается через 4 секунды.

Состояния:

- «Проверка» — результат запроса ещё не получен;
- «Онлайн» — опубликованный сайт ответил успешно;
- «Офлайн» — устройство сообщает об отсутствии сети, запрос завершился ошибкой
  или не получил успешного ответа.

Индикатор описывает доступность сайта, а не качество или тип подключения.

## Автоматические тесты

Тесты запускаются встроенным тестовым модулем Node.js с поддержкой удаления
типов TypeScript:

```powershell
npm test
```

В каталоге `tests/` находятся 18 тестов:

- `calculations.test.ts` проверяет разбор топлива, топливные формулы, ошибку при
  остатке сдачи выше остатка приёмки и отклонение от норматива;
- `dateTime.test.ts` проверяет форматы даты и времени, невозможные значения,
  длительность в пределах суток и переход на следующие сутки;
- `duration.test.ts` проверяет обычный и компактный ввод длительности, большие
  значения часов и отклонение некорректных значений.

Тесты относятся к чистым функциям и не запускают React-компоненты, браузер или
service worker.

## Текущие ограничения

- OCR ещё не реализован; кнопка отключена и помечена «Скоро».
- Нет серверной базы, авторизации, синхронизации и резервного копирования.
- Нет автоматических компонентных и сквозных браузерных тестов.
